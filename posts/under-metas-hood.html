Under Meta's hood: the health restriction, unmasked

April 17, 2026  ·  18 min read

Every health brand I've helped has landed in the same room. Ads running. Performance dying. A folder of contradictory advice from three agencies. Meta support answering with the same canned paragraph. Official docs describing what exists but never what happens. Nobody saying what actually goes on inside the box.

1. What the classifier actually reads

Meta doesn't publish the classifier's feature list. Here's the observed input priority, strongest to weakest, after watching dozens of datasets get classified, un-classified (rarely), and re-classified.

  1. Landing page body text. The single biggest signal. Headers, product descriptions, testimonial copy, FAQ sections. If your pages say “PCOS,” “diabetes,” “weight loss,” “erectile,” “CBD,” “THC,” or any named condition or prescription drug, no downstream fix compensates. The classifier reads the page. This is step one.
  2. event_source_url path. Not just the domain — the full path. /products/pcos-support is read as content. The path contributes to classification before it ever gets stripped at Tier 1. People assume Core Setup means the path was never seen. It was seen, classified against, then stripped for storage.
  3. content_name and content_category strings. Plain human-readable text. “Hormonal health / PCOS” in your category is read by the same scanner that reads your page. Same weight, maybe more — because it's explicit taxonomy.
  4. Product catalog feed. If you're using Commerce Manager or running DPA, your catalog descriptions are part of the signal pool. A clean pixel payload doesn't help if your catalog feed still calls your SKU “PCOS Relief Blend.”
  5. Event name semantics — for standard events only. Purchase, Lead, AddToCart carry pre-classified meanings. Custom events don't. But the classifier still reads the rest of the payload even when the event name is a harmless step_complete. Renaming alone moves one signal out of ten.
  6. Business Manager and ad account history. What this account has been advertising for the last several months. Not just this dataset — the account. Restrictions don't migrate wholesale across accounts, but history informs how fast classification hits a fresh dataset inside an account with prior health-vertical activity.

Inputs the classifier doesn't read in any meaningful way:

  • Which CAPI vendor shipped the request
  • Your server's IP address, geographic region, TLS version, or user-agent
  • The TLD of your domain (.com vs .health vs .io)
  • Whether you use GTM client-side, GTM server-side, or direct
  • Whether you route through Cloudflare or edge compute
  • The number of custom parameters beyond a handful (quality > quantity)
  • Creative approval status — ad review and data restriction are separate pipelines

If an agency is pitching you a fix that operates on the second list, they are selling you work on inputs the classifier doesn't weigh.

2. What each restriction tier really blocks

Meta publishes the tier names. They don't publish what each one does at the field level. Here is what actually happens to your data at each tier, based on what survives in Events Manager, what the Conversions API test tool reports back, and what the Dataset Quality API returns after a blocked payload.

Tier 1 — Core Setup

Often described as the “mild” tier. It is not mild. Core Setup is the first indicator that classification has already happened; your dataset has already been labelled health, and these strips are the consequence.

What gets stripped before ingestion:

  • event_source_url — full URL removed. Not redacted. Removed. Landing page signal goes to zero.
  • URL parameters inside payload fields — UTMs survive in reporting but not in match or optimization.
  • Custom parameters that contain URL-like strings (including referrer chains you may have passed in custom_data).

What survives:

  • Identity fields: hashed em, ph, external_id, fbp, fbc. These are the match surface, and match still works at Tier 1.
  • Standard event names and their value / currency pair.
  • IP and user agent for browser events.

What this means for optimization: you still attribute conversions. You still retarget. Lookalikes still build. But the model has lost its richest context — page, path, content taxonomy — so CPMs rise and prospecting weakens inside two to three weeks. Core Setup is survivable. It is not neutral.

Tier 2 — Standard Event Restrictions

The middle tier is where brands panic. Standard events get rejected at ingestion for specific event types. In practice this means Purchase, Lead, CompleteRegistration, Subscribe, and StartTrial lose their pre-classified meaning. They are still accepted as payloads. They are not used for campaign optimization at the standard-event layer.

What blocks:

  • Optimization against the restricted standard events for health-adjacent campaigns.
  • Dynamic product ads tied to those standard events.
  • Conversion lift studies and attribution reports at the standard-event granularity.

What still works — and this is the part nobody tells you:

  • Custom events with identical payloads. Rename Purchase to order_complete and the event ingests, attributes, and optimizes. You lose the pre-classified value meaning; you keep the signal.
  • Custom conversions built on top of those custom events. This is the path most health brands end up on without understanding why.
  • Value optimization via custom events — provided the payload is clean of condition keywords.

Tier 2 is not a ban. It is a forced migration from standard to custom event taxonomy. The brands that survive it are the ones that read the tier for what it is and rebuild the event layer; the ones that keep firing Purchase into the void lose months.

Tier 3 — Full Restrictions

The terminal tier. The dataset is flagged and demoted. This is where the folklore breaks down, because Tier 3 is not a kill switch — it is a severe optimization penalty with specific carve-outs.

What blocks:

  • All standard-event optimization for the flagged domain.
  • Dynamic product ads entirely.
  • Conversion-objective campaigns pointed at that domain — they will serve at a fraction of prior scale.
  • Advantage+ Shopping for the flagged domain.

What surprisingly survives:

  • Offline conversions uploaded through the same business, provided they reference a different, clean domain.
  • Brand-awareness and reach objectives — they don't rely on the conversion pipeline.
  • Lead Ads (the on-Meta instant form variant), because the conversion is on Meta property, not the flagged domain.
  • Another clean domain in the same business, if it is not a mirror of the flagged one.

Tier 3 is where most agencies quietly recommend a new domain. What they don't tell you: the same Business Manager can host the new domain without penalty if the content, catalog, and pixel are genuinely different. The restriction is domain-scoped. History is account-informed but not account-terminal.

3. What matters. What doesn't. A binary cheat sheet.

The single most expensive thing about this vertical is the amount of work done on inputs that don't move the needle. Here's the honest split.

Matters

  • Landing page copy. Condition names, prescription drug names, diagnostic language. Rewrite before anything else.
  • URL path segments. /pcos, /diabetes, /weight-loss — route to neutral slugs or the page-path signal keeps firing.
  • Product titles and catalog feed. Every row. Including the ones you stopped running ads against.
  • Meta title and description tags. The classifier reads the head of the document.
  • Pixel content_name, content_category, content_ids. Clean or remove.
  • Event taxonomy decision: standard vs custom. At Tier 2+, custom wins.
  • Hashing discipline. em, ph, external_id hashed; fbp, fbc never hashed. Get this wrong and match fails silently.
  • Event Match Quality. Below 6.0 and your signal is noise. Above 8.0 and the classifier has enough to read you cleanly either way.
  • Deduplication via event_id. Pixel and CAPI firing the same event without a shared event_id is double-counting, not redundancy.
  • Business verification and domain verification status. Unverified domains get classified harder and recover slower.

Doesn't matter

  • CAPI vendor choice. Stape, custom server, GTM server-side, native Shopify CAPI — all indistinguishable to the classifier.
  • Server location, CDN, TLS cipher suite. None of it is read.
  • Domain TLD. .com, .co, .health are treated identically on the content signal.
  • Number of custom parameters, beyond a sane handful. Twenty custom params don't outperform five well-chosen ones.
  • Ad creative copy — for data restriction purposes. Creative is reviewed separately by ad review. Two different pipelines. Clean creative does not un-restrict a dataset.
  • Campaign structure, budget, bidding strategy. The restriction is upstream of the auction.
  • Which browser your customers use.
  • Whether you have a Meta rep. Reps cannot lift a data restriction. The lift, when it happens, is algorithmic.

If you are being charged for work from the second list, you are paying for motion, not progress.

4. “What happens if…” — the scenarios nobody answers

These are the exact questions I get in Slack DMs from founders, answered at the mechanism level rather than the policy level.

What happens if I switch my CAPI vendor mid-flight?

Nothing, from a restriction standpoint. The classifier does not read the sender. You will see a temporary dip in Event Match Quality while the new vendor's payload shape stabilizes — usually 48 to 72 hours. If the restriction was already in place, switching vendors will not lift it. If it wasn't, switching won't cause it.

What happens if I migrate to a new domain with the same Business Manager?

The new domain starts with a clean content score but an informed account history. Classification on the new domain typically takes 10–21 days if the content is genuinely distinct. If the new domain is a reskinned clone — same product names, same category taxonomy — the classifier catches it faster than the first time, not slower. Account history accelerates recognition of known content patterns.

What happens if I remove all condition keywords from my site but leave the catalog?

You close the front door and leave the side door open. The catalog feed is a first-class input. Brands do this constantly — clean the pages, forget the feed — and then wonder why nothing changes after six weeks.

What happens if I pause all ads for 30 days?

Nothing useful. Restriction is on the dataset, not the campaign. The dataset continues receiving organic traffic events while ads are paused, and classification holds. Pausing is not a cooldown.

What happens if I send a clean payload from CAPI while the pixel still fires dirty data?

The dirty pixel wins. Both fire into the same dataset. The classifier reads the union. A clean CAPI does not cancel a dirty pixel; it dilutes it at best.

What happens if my customer's browser blocks the pixel?

CAPI carries the event. This is the expected path for about 30–40% of health-brand traffic on iOS. It is also why CAPI-only setups sometimes look healthier than pixel+CAPI — not because CAPI is cleaner, but because the pixel was leaking condition context the CAPI payload doesn't contain.

What happens if I run Lead Ads instead of website conversions?

The conversion happens on Meta's property. Your domain's classification still exists, but the optimization path sidesteps it. Lead volume is often the fastest route back to performance for Tier 3 brands, at the cost of CRM integration complexity.

What happens if I appeal the restriction?

The appeal is read by the same system that classified you, against the same inputs. If the inputs have not changed, the appeal outcome will not change. Appeals are useful only after material site, catalog, and event changes have been live long enough to be re-classified — roughly 14 days.

What happens if I use Limited Data Use mode?

LDU is a privacy control for regulated US states. It is not a health-restriction mitigation. Turning it on does not de-classify a dataset. Turning it off does not cause classification. It is orthogonal, and treating it as a lever here wastes engineering time.

What happens if I hash fbp and fbc by accident?

Match fails silently. The events ingest. They don't match. Your EMQ drops without an error message. I have seen this exact mistake cost a brand six weeks of investigation because nothing in the UI flags it.

What happens if I rename my standard event to a custom name but keep event_name strings that contain condition words?

You moved one signal and kept another. The classifier reads event names. pcos_purchase_complete is worse than Purchase was. Rename to neutral verbs: order_complete, checkout_done, signup_finished.

What happens after the January 12, 2026 attribution changes for a restricted dataset?

The 7-day view and 28-day view windows are gone. Restricted datasets lose their richer attribution alongside everyone else, but the relative disadvantage sharpens — a restricted brand is now competing on 1-day-click and 7-day-click only, against unrestricted brands with the same windows. The restriction penalty compounds with the window compression. Plan for it.

5. Timing and aging — the mysteries no doc explains

Every brand asks the same three timing questions and gets three non-answers from support. Here are the observed windows.

How long does classification take on a fresh dataset?

Between 4 and 14 days from first meaningful event volume, inside a Business Manager with prior health-vertical history. 14 to 45 days in a fresh Business Manager with no history. The range is wide because it depends on how fast the dataset crosses the classifier's volume threshold for confident labelling — roughly a few hundred events per day sustained.

Does a restriction expire?

Not on a clock. It lifts when the inputs re-classify clean. If your site, catalog, and event payloads all read as non-health for 14+ days of sustained traffic, the classifier re-evaluates. The re-evaluation is not a scheduled job; it runs against datasets that have shown enough signal change to warrant a fresh look. Sit on stale changes for 30 days and the re-evaluation may not have run yet.

How old does a Business Manager need to be?

Age is less important than history. A 6-year-old BM with a clean track record behaves like a fresh BM. A 3-month-old BM that has never triggered a restriction is in better standing than a 5-year-old BM with two prior Tier 3 domains. The system weighs behavior, not birthday.

When do changes show up?

Site copy changes: propagate to classification within 48–96 hours of being indexed. Catalog feed changes: 24–72 hours after the next feed fetch. Event payload changes: immediate on the ingestion path, but take 7–14 days to shift the aggregate classification score enough to matter.

What about the Advertiser Verification deadline?

Meta's Advertiser Verification (effective March 27, 2026) adds an identity layer on top of all of this. It does not replace or reset health classification. A verified advertiser running a classified health domain stays classified. The two systems run in parallel.

6. The support black hole — and the three surfaces that actually tell you the truth

Meta support cannot lift a health classification. That is not a policy failure; it is an architectural one. Support agents do not have a tool that says “un-restrict this dataset.” The restriction is an output of the classifier, and the classifier takes inputs, not tickets.

What support can do: confirm a dataset is restricted, confirm which tier, and escalate obvious misclassifications (a shoe brand accidentally flagged health). What support cannot do: tell you what triggered it, how to fix it, or how long until it lifts.

If you are paying a rep to pursue a support path, you are paying for a ceremony. The three surfaces that actually tell you what is happening:

  1. Events Manager → Dataset → Diagnostics tab. This is where restriction status, missing fields, and match quality degradation actually surface. The banner language is deliberately vague but the field-level diagnostics are honest.
  2. The Conversions API test tool. Send a real payload. Read the response. It will tell you which fields were accepted, which were stripped, and which were rejected. This is ground truth — not the docs, not the rep.
  3. The Dataset Quality API. Programmatic access to match rate, event coverage, and parameter quality over time. A declining curve without a corresponding campaign change is the earliest detectable signal of classification tightening. Most brands never query it. You should.

Between those three, everything the classifier is doing becomes visible enough to act on. Nothing outside them — not support tickets, not Slack groups, not Twitter, not the agency deck — will tell you more than those three surfaces already do.
